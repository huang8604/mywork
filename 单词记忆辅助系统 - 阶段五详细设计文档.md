# 单词记忆辅助系统－阶段五详细设计

## 1. 阶段目标与发布边界

系统采用单容器全栈部署：

1. 构建 Vue 静态文件；
2. FastAPI 提供 API，并在所有 API 和健康路由之后挂载 SPA；
3. SQLite 数据目录挂载到 NAS；
4. GitHub Actions 自动测试、构建并推送 GHCR 的 `latest` 与 SHA 标签；
5. NAS 运维人员**手动**进入 Portainer，Pull latest 后 Recreate。

第 5 步是明确的人工发布要求。不配置 Watchtower、Webhook、Portainer API 或脚本自动更新 NAS，也不把该流程描述为“无感部署”或“热更新”。Recreate 会产生短暂重启。

## 2. 生产镜像

### 2.1 构建原则

- Node、Python 基础镜像固定到受支持版本和镜像 digest。
- 前端必须提交 lockfile，并使用 `npm ci`。
- Python 依赖锁定版本；发布构建可使用 hash 校验。
- 添加 `.dockerignore`，排除 `.git`、`node_modules`、缓存、本地数据库和测试产物。
- 运行阶段不包含 Node、编译缓存和开发依赖。
- 容器以固定非 root UID/GID 运行，仅 `/app/data` 可写。

### 2.2 Dockerfile 结构

```dockerfile
# 实现时将基础镜像固定为“受支持版本@sha256:digest”
FROM node:<supported>-alpine@sha256:<digest> AS frontend-builder
WORKDIR /build/frontend
COPY frontend/package.json frontend/package-lock.json ./
RUN npm ci
COPY frontend/ ./
RUN npm run test && npm run build

FROM python:<supported>-slim@sha256:<digest> AS backend-runner
ENV PYTHONDONTWRITEBYTECODE=1 \
    PYTHONUNBUFFERED=1
WORKDIR /app

RUN groupadd --gid 10001 app \
    && useradd --uid 10001 --gid app --no-create-home app

COPY backend/requirements.lock ./
RUN pip install --no-cache-dir --require-hashes -r requirements.lock
COPY --chown=app:app backend/ ./
COPY --chown=app:app --from=frontend-builder /build/frontend/dist /app/static
RUN mkdir -p /app/data && chown app:app /app/data

USER 10001:10001
EXPOSE 8000
HEALTHCHECK --interval=30s --timeout=3s --start-period=20s --retries=3 \
  CMD ["python", "-c", "import urllib.request; urllib.request.urlopen('http://127.0.0.1:8000/healthz/live', timeout=2)"]
CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000", "--proxy-headers"]
```

占位的版本和 digest 必须在实现时替换，不能原样提交。

### 2.3 FastAPI 路由顺序

1. 注册 `/healthz/live`、`/healthz/ready`；
2. 注册 `/api/v1/*`；
3. 提供静态资源；
4. 最后为仅 GET/HEAD 的非 API 路径提供 SPA `index.html` 回退。

不得提前 `app.mount("/", ...)` 吞掉 API、健康检查或真实 API 404。

## 3. Portainer Stack

```yaml
# 以下镜像名、NAS 路径和域名均为占位示例，部署前必须替换为实际值。
services:
  vocab-app:
    image: ghcr.io/your-github-username/vocab-app:latest
    container_name: vocab-app
    restart: unless-stopped
    ports:
      - "8080:8000"
    volumes:
      - /share/Container/vocab-app/data:/app/data
      - /share/Container/vocab-app/secrets/api-token-pepper:/run/secrets/api-token-pepper:ro
    environment:
      DATABASE_URL: sqlite:////app/data/vocab.db
      APP_TIMEZONE: Asia/Shanghai
      PUBLIC_BASE_URL: https://words.example.com
      CORS_ORIGINS: https://words.example.com
      TRUSTED_HOSTS: words.example.com
      API_TOKEN_PEPPER_FILE: /run/secrets/api-token-pepper
      API_RATE_LIMIT_PER_MINUTE: "60"
      FORWARDED_ALLOW_IPS: 172.18.0.0/16
    healthcheck:
      test: ["CMD", "python", "-c", "import urllib.request; urllib.request.urlopen('http://127.0.0.1:8000/healthz/live', timeout=2)"]
      interval: 30s
      timeout: 3s
      retries: 3
      start_period: 20s
    security_opt:
      - no-new-privileges:true
    cap_drop:
      - ALL
```

- NAS 宿主目录预先创建，并赋予 UID/GID 10001 写权限。
- 若反向代理不在同一 Docker 网络，应由 NAS 防火墙限制 8080，仅允许可信局域网或反向代理访问。
- HTTPS、访问认证、请求体限制和必要限流由反向代理负责。
- 外部 Skill 必须通过 HTTPS 反向代理访问 API，并由应用自身验证 Bearer Token 和 scope；CORS 不适用于服务端到服务端调用，不能把 CORS 当作认证。
- Token pepper 文件权限限制为运行 UID 可读；各 Skill 的明文 Token 只保存在对应 Skill 的 Secret 配置中，不写入 Compose、镜像、仓库或日志。
- 仅信任实际反向代理网段传入的 `X-Forwarded-*`，代理应删除客户端伪造的同名请求头。
- 是否启用只读根文件系统在实现时验证；临时文件使用 tmpfs，SQLite 数据卷保持可写。

## 4. 数据迁移、备份与恢复

### 4.1 更新前

1. 记录当前运行镜像的 SHA 标签和数据库 schema 版本。
2. 使用 SQLite 在线备份 API 或停写后复制，生成时间戳备份。
3. 执行 `PRAGMA integrity_check` 并确认备份可读。
4. 至少保留最近若干版本，并将关键备份复制到不同物理存储。

不能只把 Volume 挂载等同于备份。

### 4.2 启动迁移

- 新容器先执行版本化迁移；失败则 readiness 不通过。
- 破坏性迁移必须拆成向前兼容的多步发布。
- 回滚旧镜像前检查其是否兼容当前 schema；不兼容时先恢复更新前备份。

### 4.3 恢复演练

定期在临时目录恢复备份，启动同版本镜像并验证单词数量、流水数量和 `word_stats` 重建一致性。文档记录期望 RPO/RTO；个人 NAS 建议至少做到更新前备份和周期备份。

## 5. GitHub Actions

### 5.1 触发和权限

- 触发：push 到 `main`；Pull Request 只测试、不推送镜像。
- 权限最小化：`contents: read`、发布 job 才有 `packages: write`。
- 第三方 Actions 固定完整 commit SHA，并由依赖更新工具维护。

### 5.2 门禁

发布 job 必须依赖：

1. 后端格式、静态检查和测试；
2. 前端类型检查、单元测试和构建；
3. OpenAPI 与前端类型一致性检查；
4. Docker 构建及容器 smoke test；
5. 依赖和镜像漏洞扫描。
6. 外部 Skill 契约测试：发现、OpenAPI、scope、Token 撤销、生成幂等和批量回录原子性。

全部通过后，Docker metadata 同时产生：

- `latest`：仅指向 main 最新成功构建；
- `sha-<commit>`：不可变，用于审计和回滚。

SBOM、镜像签名和构建证明作为推荐增强项。

## 6. NAS 人工更新流程

1. 在 GitHub Actions 确认目标提交的测试和镜像发布成功。
2. 记录对应的 `sha-<commit>`。
3. 按第 4 节完成数据库备份及校验。
4. 进入 Portainer 的 `vocab-app`。
5. 手动 Pull `ghcr.io/.../vocab-app:latest`。
6. 手动 Recreate，沿用原环境变量和数据卷。
7. 等待容器变为 healthy，访问 `/healthz/ready`，再执行登录、首页以及使用专用只读 Token 调用 capabilities 的 smoke test。完整 Skill 生成/回录流程在 CI 的临时数据库中验证，避免污染生产数据。
8. 记录实际运行 SHA、更新时间和操作者。

本流程不增加 NAS 自动拉取或自动重启机制。

## 7. 人工回滚

1. 在 Portainer 将镜像改为上一个已知正常的 `sha-<commit>`，不要依赖漂移的 `latest`。
2. 检查数据库迁移兼容性；必要时停止容器并恢复更新前备份。
3. Recreate 后等待 healthy，并执行 smoke test。
4. 记录失败版本、原因和恢复结果。

## 8. Definition of Done

- 镜像构建可复现，运行容器为非 root，健康检查生效。
- API、SPA 深链和健康路由互不吞噬。
- CI 测试通过后才推送 `latest` 与 SHA 标签。
- 数据卷权限、迁移、备份、恢复和按 SHA 回滚均完成演练。
- 反向代理启用 HTTPS、访问控制和受限来源。
- 外部 Skill Token、pepper、scope、限流和审计配置通过生产安全检查；公网无法绕过反向代理直连 8080。
- CI 在临时数据库中通过“Skill 生成复习表 → 创建轮次 → 批量录入三态”的端到端测试。
- Portainer 人工 Pull latest + Recreate 流程有明确检查表，且不存在额外自动部署链路。
